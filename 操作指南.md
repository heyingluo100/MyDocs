# 操作指南（从零开始）

> 本指南把"实操主线"放在最前面：拉代码 → 本地跑起来 → 推到 GitHub → 部署上线 → 日常更新文档。
>
> **如果你完全没用过 GitHub、没装过 Node.js / Git** —— 先跳到「**第五章 · 首次使用前的准备**」把账号注册、软件安装搞定，然后回到第一章按主线走。
>
> **如果你已经有 GitHub 账号、本地装了 Node 和 Git** —— 直接从第一章开始读。

---

## 一、整体链路

整个项目的工作流是这条链：

```
本地电脑                    GitHub                   Netlify / Vercel / GitHub Pages
┌─────────┐    git push    ┌────────┐    自动构建    ┌──────────────────┐
│ 写文档  │ ─────────────→ │ 仓库   │ ────────────→ │ 你的网站（公开） │
│ npm run │                │ 代码 + │                │  xxx.netlify.app  │
│ build   │                │ 文档   │                │  xxx.vercel.app   │
└─────────┘                └────────┘                └──────────────────┘
```

- **你只在本地动文档**（往 `content/` 里加 / 改文件）
- **每次想发布就推一下** GitHub
- **网站会自动重新部署**，1-2 分钟后访问者就能看到新内容

整条链一次配好后，日后只剩一条命令：`npm run build && git add . && git commit -m "更新" && git push`。

## 二、核心链路（实操）

> 第一次走这章会比较慢（要拉代码、配关联、首次推送、配部署）。配好之后日常更新只要做 2.4「日常推送」那一步。

### 2.1 拉代码到本地

打开终端，逐行输入：
```
git clone https://github.com/heyingluo100/test1.git my-docs
cd my-docs
```

> 这会把项目模板下载到你电脑上的 `my-docs` 文件夹里。
>
> **没装 Git？** 看 5.5 节装一下再回来。

### 2.2 安装依赖 + 本地预览

```
npm install
```

等待安装完成（可能需要几分钟，看到没有红色报错就行）。

> **遇到红色报错怎么办？** 别慌，把报错信息**完整复制**（从第一行 `npm error` 或 `npm ERR!` 一直到最后），丢给 AI 工具就行：
>
> - **DeepSeek**：https://chat.deepseek.com
> - **通义千问**：https://tongyi.aliyun.com
> - **Monica**：https://monica.im
>
> 提问模板：
> ```
> 我在 Windows / Mac 上跑 npm install 报错，请帮我看看怎么解决，下面是完整报错：
>
> [把报错全部粘贴到这里]
> ```
>
> AI 会告诉你原因（一般是网络慢、Node 版本不对、权限不够等）和具体的修复命令。照它说的做，再 `npm install` 重试一遍。

然后本地启动：
```
npm run dev
```

浏览器打开终端显示的地址（一般是 `http://localhost:5173`）就能看到网站。文件有变动会自动更新，不用手动刷新。

> **没装 Node.js？** 看 5.4 节装一下再回来。
>
> Windows 用户可以直接双击项目根目录的 `start.bat`，效果一样。

### 2.3 创建 content 目录 + 写第一篇文档

`content/` 是你放所有文档的地方，**不会上传到 GitHub**（在 `.gitignore` 里），属于你自己的私有内容。

**方式一：图形界面（推荐新手）**

打开项目文件夹（刚才 clone 下来的 `my-docs`），在里面**右键 → 新建文件夹**，把文件夹命名为 `content`。

**方式二：命令行**
```
mkdir content
```

两种方式效果完全一样，挑顺手的就行。

然后在 `content/` 里创建一个**子文件夹**（就是分类标签，同样**右键新建**就行），把你的文档放进去。例如：

```
content/
└── 公告/
    └── 欢迎.md     ← 内容随便写点
```

刷新浏览器就能看到这篇文档出现在网站上。

### 2.4 关联到你自己的 GitHub 仓库

刚才 clone 下来的代码默认连到模板仓库，你要把它改成连到你自己的仓库。

**先建仓库**：
1. 登录 GitHub，点右上角 **+** → **New repository**
2. 填写：
   - Repository name：随便取个名字（如 `my-docs`）
   - 选 **Private**（强烈建议私有，避免别人 fork 拉你的 `.lock` 哈希做撞库）
   - **不要**勾选任何初始化选项（不要勾 README、.gitignore）
3. 点 **Create repository**
4. 复制页面上显示的仓库地址（如 `https://github.com/你的用户名/my-docs.git`）

**改本地关联**：
```
git remote set-url origin https://github.com/你的用户名/my-docs.git
```

> 把上面的地址换成你刚复制的那个。
>
> **没注册 GitHub？** 看 5.1 节再回来。

### 2.5 首次构建并推送

```
npm run build
git add .
git commit -m "首次提交"
git push -u origin master
```

> 如果提示输入用户名密码，输入你的 GitHub 用户名和 **Personal Access Token**（不是登录密码）。
>
> 获取 Token：GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token，勾选 `repo` 权限。

推送成功后，去 GitHub 仓库页面应该能看到代码已经上传。

### 2.6 部署到 Netlify（推荐）

**为什么推荐 Netlify**：国内可直接登录管理后台，不用翻墙；免费套餐每月 15 GB 流量对个人网站完全够用。

1. 打开 https://app.netlify.com
2. 点 **Add new site** → **Import an existing project**
3. 选 **Deploy with GitHub**，授权后找到你刚推送的仓库（`my-docs`）
4. 构建配置（如果没自动填好就手动填）：
   - **Build command**：`npm run build`
   - **Publish directory**：`dist`
5. 点 **Deploy site**，等 1-2 分钟，部署完成
6. Netlify 会给你一个网址（如 `https://random-name-xxx.netlify.app`），这就是你的网站
7. 想改这个二级域名：进站点 → **Site configuration** → **Change site name**，改成你喜欢的（如 `my-docs.netlify.app`）

以后每次 `git push`，Netlify 自动重新部署。

> **关于 SPA 路由**：项目里已经放了 `public/_redirects`（一行 `/*  /index.html  200`），Netlify 自动识别，刷新内页不会出现 404。这个文件不用动。
>
> **没注册 Netlify？** 看 5.2 节再回来。

### 2.7 部署到 Vercel（备选）

1. 打开 https://vercel.com/dashboard
2. 点 **Add New...** → **Project**
3. 找到你刚推送的仓库（`my-docs`），点 **Import**
4. 设置页面保持默认，直接点 **Deploy**
5. 等 1-2 分钟，部署完成
6. Vercel 给你一个网址（如 `https://my-docs-xxx.vercel.app`），这就是网站地址（前缀可改，后缀 `.vercel.app` 不能改）

以后每次 `git push`，Vercel 自动重新部署。

> **没注册 Vercel？** 看 5.3 节再回来。

### 2.8 部署到 GitHub Pages（备选）

1. 打开你的 GitHub 仓库页面
2. 点 **Settings** → 左侧 **Pages**
3. Source 选择 **GitHub Actions**
4. 项目已经包含了自动部署的配置文件（`.github/workflows/deploy.yml`），推送代码后会自动构建部署
5. 等几分钟，网站地址为 `https://你的用户名.github.io/my-docs/`

### 2.9 自定义域名

三种部署平台都支持绑定自己的域名（如 `docs.example.com`），步骤大同小异。

#### Netlify
1. 进入站点 → **Domain management** → **Add a domain**
2. 输入域名 → **Verify** → **Add domain**
3. 去域名注册商后台加 DNS 记录：
   - 子域名（如 `docs.example.com`）：**CNAME** 记录，值为 Netlify 显示的目标（通常是 `<你的站点>.netlify.app`）
   - 根域名（如 `example.com`）：**A** 记录，值为 Netlify 显示的 IP（**以 Netlify 后台实时显示的值为准**）
4. 等 DNS 生效（几分钟到几小时），回 Netlify 点 **Verify DNS configuration**，自动配 HTTPS

#### Vercel
1. 进入项目 → **Settings** → 左侧 **Domains**
2. 输入域名 → **Add**
3. 去域名注册商后台加 DNS 记录：
   - 子域名：**CNAME** 记录，值为 `cname.vercel-dns.com`
   - 根域名：**A** 记录，值为 `76.76.21.21`
4. 等 DNS 生效，Vercel 自动配 HTTPS

#### GitHub Pages
1. 仓库 **Settings** → **Pages** → **Custom domain** 输入域名 → **Save**
2. 域名注册商加 CNAME 记录：`你的用户名.github.io`
3. 等生效，GitHub 自动配 HTTPS

> **没有域名？** 不用买，直接用平台给的免费二级域名就行。

### 2.10 修改网站名称

网站默认叫「文档库」，改名需要改 3 个地方：

| 文件 | 位置 | 改什么 |
|------|------|--------|
| `index.html` | 第 9 行 `<title>` | 浏览器标签页上显示的标题 |
| `src/components/AppHeader.vue` | 搜索「文档库」 | 网站左上角显示的名称 |
| `src/App.vue` | 搜索「Personal Docs」 | 网站底部的版权文字 |

改完后在终端执行下面这条命令，就会自动构建并推送到部署平台：

```
npm run build && git add . && git commit -m "更新网站名称" && git push
```

等 1-2 分钟，Netlify / Vercel / GitHub Pages 自动重新部署，刷新网站就能看到新名称。

> 邀请码验证页面（`src/components/InvitationGate.vue`）上也有一行「Personal Docs」，如果开了全站锁也一起改掉。

## 三、日常使用

### 3.1 添加 / 修改文档

1. 在 `content/` 下创建文件夹（文件夹名 = 分类标签）
2. 把 `.docx`、`.md` 或 `.txt` 文件拖进去
3. 完成

**目录结构示例**：
```
content/
├── 工作/                ← 分类「工作」
│   ├── 周报.docx        ← Word 文档
│   └── 项目总结.md      ← Markdown 文档
├── 学习/                ← 分类「学习」
│   └── 笔记.docx
├── 公告/
│   ├── 欢迎.md          ← 单独文章
│   └── 版本记录/        ← 合集（子文件夹 = 合集）
│       ├── v0.1.md
│       └── v0.2.md
└── P3/
    └── P345联动合集/    ← 跨标签合集
        ├── .tags        ← 内容: P4\nP5（额外标签）
        ├── 01-第一章.md
        └── 02-第二章.md
```

### 3.2 合集

在标签文件夹下创建**子文件夹**，即为合集。合集内文章按文件名排序（建议用 `01-`、`02-` 数字前缀控制顺序）。

### 3.3 多标签（标签穿插）

文章或合集需要出现在多个标签下时，创建 `.tags` 文件指定额外标签：

- **合集多标签**：在合集文件夹内新建 `.tags`（文件名就叫 `.tags`），每行写一个额外标签名
- **单独文章多标签**：在同级目录新建 `文件名.tags`（如 `周报.tags` 对应 `周报.docx`），每行写一个额外标签名

**注意**：合集用的是 `.tags`（不带前缀），单独文章用的是 `文件名.tags`（带文件名前缀），别搞混。

### 3.4 本地预览

```
npm run dev
```

浏览器打开显示的地址即可预览。文件有变动自动更新，无需手动刷新。

### 3.5 发布上线

```
npm run build && git add . && git commit -m "更新文档" && git push
```

推送后 Netlify / Vercel / GitHub Pages 自动部署，1-2 分钟后网站更新。

## 四、进阶配置

### 4.1 邀请码系统

网站支持两层邀请码保护，默认关闭，按需开启。

#### 生成邀请码

```
npm run gen-code
```

输出邀请码（明文，发给受邀用户）和哈希值（填入配置）。也可以自定义码：

```
npm run gen-code "你想要的码"
```

#### 开启全站锁

全站锁开启后，访问任何页面都必须先输入邀请码。

1. 运行 `npm run gen-code`，记下邀请码和哈希值
2. 打开 `src/config/invitation.js`，修改：
```js
global: {
  enabled: true,
  codeHash: '粘贴哈希值到这里',
}
```
3. 重新构建发布
4. 把邀请码发给受邀用户

#### 锁定单篇文章

单篇锁可以给特定文章设独立邀请码，每篇码可以不同。

1. 运行 `npm run gen-code`，记下邀请码和哈希值
2. 在文章旁创建同名 `.lock` 文件，内容为哈希值：
```
content/
├── 工作/
│   ├── 机密报告.docx
│   └── 机密报告.lock    ← 内容为哈希值（一行）
```
3. 打开 `src/config/invitation.js`，设 `article.enabled: true`
4. 重新构建发布
5. 该文章的专属邀请码发给特定用户

#### 锁定整个合集

在合集文件夹内放 `.lock` 文件（和 `.tags` 同级），合集内所有文章自动锁住，共享同一个邀请码。

1. 运行 `npm run gen-code`，记下邀请码和哈希值
2. 在合集文件夹内创建 `.lock` 文件，内容为哈希值：
```
content/
├── 工作/
│   └── 机密合集/
│       ├── .lock            ← 内容为哈希值，整个合集一把锁
│       ├── 01-第一章.md
│       └── 02-第二章.md
```
3. 打开 `src/config/invitation.js`，设 `article.enabled: true`
4. 重新构建发布

如果合集内某篇文章还有自己的 `.lock` 文件（如 `01-第一章.lock`），那篇文章优先用自己的码，其余用合集的码。

#### 关闭邀请码

把 `src/config/invitation.js` 里对应的 `enabled` 改为 `false`，重新构建即可。

> ⚠️ **关键提醒：单篇锁的总闸/开关层级**
>
> `article.enabled` 是**总闸**，`.lock` 文件是**单篇开关**。两者关系：
>
> - **总闸开**（`enabled: true`）→ 才识别 `.lock` 文件，按邀请码加密保护
> - **总闸关**（`enabled: false`）→ 所有 `.lock` 文件失效，构建时**跳过加密、文章按明文打包**，UI 不显示锁图标
>
> 所以**关闭总闸 = 锁定文章变公开**：你过去给文章设的 `.lock` 在前端、在 `articles.json` 里全部以**明文**形式可读，部署后任何人能看到。如果只是想让某几篇暂时公开，建议**保留 `.lock` + 总闸开**，单独删除那几篇的 `.lock`，比关总闸安全。
>
> 全站锁（`global.enabled`）独立，不受这个层级影响。

#### 注意

- 邀请码验证后 **7 天有效**，到期会自动重新锁上（卡片图标变回暗酒红闭锁，需重新输入）
- 全站锁和单篇锁可以同时开启（双层保护）
- 配置文件中只存哈希值，不存明文码
- `.lock` 文件里也是哈希值，可以安全提交到 Git
- 锁定文章的内容使用 **AES-256-GCM 加密**存储，密钥由邀请码经 **PBKDF2-SHA256 迭代 200000 次 + 每篇独立 salt** 派生（v2 协议），即使有人拿到 articles.json 离线撞库，每次尝试也要完整跑一遍 PBKDF2，比单次 SHA-256 慢 20 万倍，短邀请码也大幅延缓破解
- 客户端还内置**暴力破解防护**：连续输错邀请码每次会指数级延迟（1s→2s→4s→8s），第 5 次错后该篇文章/全站锁锁定 5 分钟。每篇独立计数，A 文章被锁不影响 B 文章
- 未加锁文章是 Base64 编码的公开内容，技术上可被读取——重要内容请加 `.lock` 才有真正保护
- 网站还做了 27 个 AI 爬虫的 robots.txt 黑名单 + noai meta 声明，合规 AI 公司不会抓取/训练你的内容
- **邀请码长度建议 8 位以上**：虽然系统已用 PBKDF2 加密 + 频率限制，但极短的邀请码（4-5 位）仍存在被穷举的可能。`npm run gen-code` 默认生成 12 位带分隔符的码（如 `Abcd-eFgh-2345`），强度足够

### 4.2 文档置顶

支持将重要文档或合集固定在列表顶部，支持全局置顶和标签内置顶。

#### 置顶单篇文章

在文章旁创建同名 `.pin` 文件：
```
content/
├── 工作/
│   ├── 重要报告.docx
│   └── 重要报告.pin    ← 置顶标记
```

#### 置顶整个合集

在合集文件夹内创建 `.pin` 文件（不带前缀，跟 `.tags` 一致）：
```
content/
├── 工作/
│   └── 重要合集/
│       ├── .pin           ← 整个合集置顶
│       ├── 01-第一章.md
│       └── 02-第二章.md
```

#### 合集内文章置顶

在合集文件夹内用 `文件名.pin` 指定合集内部的置顶文章：
```
content/
├── 工作/
│   └── 某合集/
│       ├── 01-第一章.md
│       ├── 01-第一章.pin  ← 合集内置顶
│       └── 02-第二章.md
```

#### .pin 文件内容格式

| 内容 | 效果 |
|------|------|
| 空文件 | 全局置顶，权重 0 |
| `1` | 全局置顶，权重 1（数字越小越靠前） |
| `2 工作` | 仅在「工作」标签内置顶，权重 2 |

多篇置顶时按权重数字从小到大排列，全局置顶优先于标签内置顶。

#### 取消置顶

删掉对应的 `.pin` 文件，重新构建即可。

### 4.3 作者的话

可以给每篇文章添加一段「作者的话」，以引用卡片样式显示在文章开头或结尾。

#### 添加作者的话

在文章旁创建同名 `.note` 文件，内容支持 Markdown：
```
content/
├── 工作/
│   ├── 周报.docx
│   └── 周报.note    ← 作者的话
```

#### .note 文件内容格式

**放在文章开头**（默认）：
```
这是作者想说的话，支持 **Markdown** 格式。
可以写多行。
```

**放在文章结尾**：
```
bottom
这是作者想说的话，放在文章最后面。
```

**放在文章开头**（显式指定）：
```
top
这段话放在文章最前面。
```

规则：
- 第一行如果是 `top` 或 `bottom`，则作为位置指令（不会显示在内容中）
- 如果第一行不是 `top`/`bottom`，整个文件都是内容，默认位置为 `top`（开头）
- 每篇文章的 `.note` 内容独立，不会重复出现在其他文章

#### 删除作者的话

删掉对应的 `.note` 文件，重新构建即可。

## 五、首次使用前的准备（如果你是新手）

> 如果你已经有 GitHub 账号、本地装了 Node.js 和 Git，**这一章可以跳过**，回到第一章按主线走。

### 5.1 注册 GitHub 账号

GitHub 是一个代码托管平台，你的网站代码会存在这里。

1. 打开 https://github.com
2. 点击右上角 **Sign up**
3. 输入邮箱、设置密码、选择用户名
4. 完成邮箱验证
5. 登录后你会看到 GitHub 首页

> **用什么邮箱都行**：QQ 邮箱、Gmail、163、Outlook 都可以，不需要专门注册新邮箱。GitHub 不支持用 Google / QQ 账号一键登录注册——它只是收个邮箱地址而已。

### 5.2 注册 Netlify 账号（推荐部署方式）

Netlify 是一个免费的网站托管平台，推送代码后自动部署。**国内可直接登录管理后台，不用翻墙**，免费额度（每月 15 GB 流量）对个人网站完全够用。

1. 打开 https://www.netlify.com
2. 点击 **Sign up**
3. 选择 **GitHub**（用 GitHub 账号登录）
4. 授权 Netlify 访问你的 GitHub
5. 完成

### 5.3 注册 Vercel 账号（备选）

Vercel 是另一个免费的网站托管平台，功能与 Netlify 相近。如果 Netlify 不顺手可以备选用它。

1. 打开 https://vercel.com
2. 点击 **Sign Up**
3. 选择 **Continue with GitHub**（用 GitHub 账号登录）
4. 授权 Vercel 访问你的 GitHub
5. 完成

> 三种部署方式（Netlify / Vercel / GitHub Pages）只需选一种即可，本指南推荐 Netlify。如果暂时只想试 Netlify，5.3 可以先跳过。

### 5.4 安装 Node.js

Node.js 是运行本项目的环境。

1. 打开 https://nodejs.org
2. 下载 **LTS 版本**（左边那个绿色按钮）
3. 双击安装，一路点「下一步」即可
4. 验证安装：打开终端（Windows 按 `Win+R`，输入 `cmd` 回车），输入：
```
node -v
```
看到版本号（如 `v20.x.x`）就成功了。

> **什么是终端？** 就是一个黑色/白色的命令输入窗口。Windows 叫「命令提示符」或「PowerShell」，Mac 叫「终端」。后续所有命令都在这里输入。

### 5.5 安装 Git + 配置身份

#### 5.5.1 安装 Git

Git 是代码版本管理工具，用来上传代码到 GitHub。

1. 打开 https://git-scm.com
2. 下载安装，一路点「下一步」
3. 验证安装：终端输入：
```
git -v
```
看到版本号就成功了。

#### 5.5.2 配置 Git 身份（首次使用）

Git 需要知道"是谁在操作"，这样每次提交代码都会记录你的名字。这不是注册账号，只是给本地电脑设一个标识。

终端依次输入下面两行命令（**把引号里的中文换成你自己的信息**）：
```
git config --global user.name "你的名字"
git config --global user.email "你的邮箱@example.com"
```

**举个例子**，如果你叫张三，邮箱是 zhangsan@qq.com，就输入：
```
git config --global user.name "zhangsan"
git config --global user.email "zhangsan@qq.com"
```

> 这个名字和邮箱不需要和 GitHub 账号一模一样，但建议保持一致，方便辨认。设置一次就行，以后不用再设。

**查看当前电脑绑定的账号**：如果不确定之前设过没有，终端输入：
```
git config --global user.name
git config --global user.email
```
会显示当前设置的名字和邮箱。如果没有任何输出，说明还没设置，按上面的步骤设一下就行。

## 六、常用操作速查

| 我想要... | 怎么做 |
|-----------|--------|
| 添加文档 | 往 `content/分类名/` 里拖入 `.docx`、`.md` 或 `.txt` 文件 |
| 新建分类 | 在 `content/` 下新建一个文件夹 |
| 新建合集 | 在分类文件夹下新建子文件夹，放入文档 |
| 多标签 | 在合集文件夹内或文件旁新建 `.tags` 文件，每行写一个额外标签 |
| 删除文档 | 删除 `content/` 里对应的文件 |
| 删除分类 | 删除 `content/` 里对应的文件夹 |
| 置顶文章 | 在文件旁创建 `文件名.pin`（详见 4.2） |
| 置顶合集 | 在合集文件夹内创建 `.pin`（详见 4.2） |
| 取消置顶 | 删掉对应的 `.pin` 文件 |
| 添加作者的话 | 在文件旁创建 `文件名.note`（详见 4.3） |
| 开启邀请码 | 详见 4.1 |
| 本地预览 | 终端运行 `npm run dev` |
| 发布上线 | `npm run build && git add . && git commit -m "更新文档" && git push` |
| 拉取最新 | 终端运行 `git pull` |

## 七、换电脑

### 情况 A：全新电脑
1. 安装 Node.js + Git（见第五章）
2. 终端执行：
```
git clone https://github.com/你的用户名/你的仓库名.git
cd 你的仓库名
npm install
mkdir content
```
3. 把你的 `content/` 文件夹复制过来
4. 正常操作即可

### 情况 B：另一台电脑已经推送过更新
```
cd 你的仓库名 && git pull && npm install
```
拉取最新代码后继续操作。

### 重要：备份 content/ 文件夹
`content/` 不会上传到 GitHub，换电脑时需要**手动拷贝**这个文件夹（U盘、网盘等）。这是你唯一需要自己备份的东西。

## 八、注意事项

1. **只操作 `content/` 目录**，其他文件不要动
2. 支持的文件格式：`.docx`（Word）、`.doc`（旧版 Word）、`.md`（Markdown）、`.txt`（纯文本）、`.pdf`（附件预览）
3. `content/` 里的原始文档**不会上传到 GitHub**，别人看不到源文件
4. 访客只能通过网站浏览，无法复制、抓取文字内容
5. 每次更新文档后记得执行「发布上线」命令
6. 特殊文件说明：
   - `.tags` → 多标签标记
   - `.lock` → 邀请码锁定
   - `.pin` → 置顶标记
   - `.note` → 作者的话
